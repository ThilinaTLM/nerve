# Message Envelope

Every Nerve Protocol v1 message uses a common envelope. The envelope provides versioning, routing, correlation, validation, and observability metadata. The payload schema is selected by `kind`.

## Canonical shape

```ts
type NerveMessage<TData = unknown> = {
  protocol: "nerve";
  version: 1;
  id: string;
  kind: NerveMessageKind;
  ts: string;
  source?: PeerDescriptor;
  target?: PeerDescriptor;
  correlationId?: string;
  causationId?: string;
  traceId?: string;
  replyTo?: string;
  requiresAck?: boolean;
  meta?: Record<string, unknown>;
  data: TData;
};
```

A JSON-encoded message example:

```json
{
  "protocol": "nerve",
  "version": 1,
  "id": "msg_01JZ7F7G8A0W4X0Q6Y6M9NE5AQ",
  "kind": "heartbeat",
  "ts": "2026-06-26T12:00:00.000Z",
  "data": {
    "sessionId": "ses_01JZ7F7E98HPJQZ0VHP19N8V8R",
    "latestSeq": 12042
  }
}
```

## Required fields

### `protocol`

`protocol` MUST be the string `"nerve"` for Nerve Protocol v1 messages.

Receivers MUST reject messages with any other value unless a transport binding explicitly supports protocol multiplexing.

### `version`

`version` MUST be the number `1`.

Receivers MUST reject messages with unsupported major versions. A receiver MAY respond with an `error` message using code `PROTOCOL_VERSION_UNSUPPORTED` if the session is established enough to send an error.

### `id`

`id` is a unique message identifier generated by the sender.

Requirements:

- `id` MUST be unique within a sender's process lifetime.
- `id` SHOULD be globally unique enough for logs and correlation.
- `id` SHOULD use the `msg_` prefix.
- `id` MUST NOT encode secrets or personally sensitive content.

The recommended format is a Nerve/ULID-style identifier such as `msg_01JZ7F7G8A0W4X0Q6Y6M9NE5AQ`.

### `kind`

`kind` selects the message schema for `data`.

Requirements:

- `kind` MUST be a non-empty ASCII string.
- `kind` SHOULD use lowercase dot-separated names.
- Control messages SHOULD use short nouns or noun phrases, such as `hello`, `welcome`, `heartbeat`, `ack`, `error`.
- Event stream messages SHOULD use grouped names, such as `event.batch`, `replay.request`, `flow.update`.
- Domain request/response messages SHOULD use protocol kinds `request` and `response` with domain method names inside `data.method`.

### `ts`

`ts` is the sender timestamp in ISO 8601 UTC format.

Requirements:

- `ts` MUST be parseable as a date-time string.
- `ts` MUST use UTC when generated by Nerve components.
- Receivers MUST NOT rely on peer clocks for ordering. Event `seq` values and message causal relationships are authoritative for ordering.

### `data`

`data` is the message payload.

Requirements:

- `data` MUST be present.
- `data` MUST conform to the schema selected by `kind`.
- For messages with no payload, `data` MUST be `{}` rather than omitted or `null`.

## Optional fields

### `source`

`source` describes the sender.

```ts
type PeerDescriptor = {
  role: "orchestrator" | "ui" | "desktop" | "cli" | "agent" | "unknown";
  id?: string;
  name?: string;
  instanceId?: string;
};
```

`source` SHOULD be included in `hello`, `welcome`, and error/log-relevant messages. It MAY be omitted from high-frequency messages once the session is established.

### `target`

`target` describes the intended recipient. It is OPTIONAL for point-to-point connections where the transport already identifies the recipient.

`target` is useful for future multiplexed transports or brokered IPC.

### `correlationId`

`correlationId` links messages that are part of the same logical operation.

Examples:

- a `request` and its `response`;
- a `replay.request`, `replay.started`, zero or more `event.batch` messages, and `replay.complete`;
- a `flow.update` caused by a specific slow-client condition.

For request/response flows, the responder SHOULD set `correlationId` to the request message `id` unless a higher-level operation correlation ID already exists.

### `causationId`

`causationId` identifies the message or event that directly caused this message.

Example: a `replay.complete` message may set `causationId` to the `replay.request` message ID.

### `traceId`

`traceId` identifies a broader trace across logs, HTTP calls, WebSocket messages, and agent operations.

Requirements:

- `traceId` MUST NOT contain secrets.
- Peers SHOULD preserve `traceId` across protocol boundaries when safe.
- If an HTTP request creates a WebSocket-visible event, the event batch metadata MAY include the same `traceId`.

### `replyTo`

`replyTo` identifies the message that this message answers. It is mainly useful for `response` and `error` messages.

`replyTo` SHOULD be set when a message is a direct response to a specific request.

### `requiresAck`

`requiresAck` is an advisory flag indicating that the sender expects a protocol acknowledgement or domain-level response.

For event streaming, durable event acknowledgement is determined by `ack` cursors, not by `requiresAck` on every `event.batch`. Implementations SHOULD NOT set `requiresAck` on high-frequency event batches unless a future capability defines per-message ack behavior.

### `meta`

`meta` is optional metadata for diagnostics, feature flags, and non-normative hints.

Requirements:

- `meta` MUST be a JSON object if present.
- `meta` MUST NOT contain secrets.
- Receivers SHOULD ignore unknown `meta` fields.
- Senders SHOULD keep `meta` small.
- Large diagnostic data SHOULD be referenced by ID or URL instead of embedded.

Suggested common metadata keys:

```ts
type CommonMeta = {
  clientVersion?: string;
  serverVersion?: string;
  transport?: "websocket" | "http" | "webtransport" | "electron-ipc" | "pipe" | "sse";
  encoding?: "json" | "messagepack" | "cbor";
  compression?: "none" | "gzip" | "br";
  attempt?: number;
  debugLabel?: string;
};
```

## Message kinds

Version 1 defines these message kinds.

| Kind | Direction | Purpose |
| --- | --- | --- |
| `hello` | client → orchestrator | Start a session, advertise capabilities, request resume. |
| `welcome` | orchestrator → client | Accept session, select capabilities, report cursors and limits. |
| `ready` | client → orchestrator | Signal that the client is ready to receive live traffic. |
| `heartbeat` | either | Liveness and lightweight cursor/status exchange. |
| `goodbye` | either | Graceful session shutdown. |
| `error` | either | Protocol, validation, auth, replay, or flow-control error. |
| `event.batch` | orchestrator → client | Deliver one or more ordered domain events. |
| `ack` | client → orchestrator | Acknowledge processed durable event cursors. |
| `replay.request` | client → orchestrator | Request replay from a cursor. |
| `replay.started` | orchestrator → client | Mark the start of a replay range. |
| `replay.complete` | orchestrator → client | Mark replay completion and final cursor. |
| `replay.unavailable` | orchestrator → client | Report that requested replay cannot be satisfied. |
| `flow.update` | either | Communicate backpressure/degradation/resync state. |
| `request` | either | Optional protocol-enveloped RPC-style request. |
| `response` | either | Optional protocol-enveloped RPC-style response. |

Transport bindings or capabilities MAY define additional kinds. Unknown message kinds are handled as described in [Unknown messages and fields](#unknown-messages-and-fields).

## Payload selection

Receivers MUST validate the envelope before interpreting `data`. After envelope validation, receivers MUST validate `data` using the schema for `kind`.

Suggested validation order:

1. Parse the transport frame/body as JSON.
2. Validate that the value is an object.
3. Validate required envelope fields.
4. Validate `protocol` and `version`.
5. Validate `kind`.
6. Validate `data` against the kind-specific schema.
7. Apply authorization/policy checks for the operation or event.
8. Dispatch to session, stream, or domain handlers.

## Encoding

### Baseline JSON

All Nerve Protocol v1 implementations MUST support UTF-8 JSON encoding.

JSON rules:

- Messages MUST encode to a single JSON object.
- `undefined`, `NaN`, `Infinity`, functions, and cyclic structures are invalid.
- Binary domain content SHOULD be transferred out-of-band or encoded using domain-specific mechanisms such as base64 only when unavoidable.
- Timestamps MUST be strings.
- Integers such as event `seq` MUST be safe JavaScript integers.

### Optional binary encodings

Future capabilities MAY negotiate encodings such as MessagePack or CBOR.

A binary encoding capability MUST define:

- canonical mapping of the envelope;
- how unknown fields are preserved or ignored;
- maximum message size;
- content-type or transport subprotocol naming;
- error behavior when a peer sends an unsupported encoding.

Until such a capability is specified and negotiated, senders MUST use JSON.

## Size limits

A transport binding MUST define maximum message size. The WebSocket v1 binding SHOULD use these defaults unless implementation measurements justify different values:

| Limit | Recommended default |
| --- | ---: |
| Maximum encoded message size | 4 MiB |
| Maximum `event.batch` encoded size | 1 MiB target, 4 MiB hard cap |
| Maximum events per batch | 500 durable events or 2,000 transient/coalesced events |
| Maximum metadata size | 16 KiB |
| Maximum error message text | 8 KiB |

Senders SHOULD split large event streams into multiple `event.batch` messages. Receivers MUST reject messages that exceed configured hard caps with `MESSAGE_TOO_LARGE` when possible.

## IDs and prefixes

Recommended prefixes:

| Entity | Prefix |
| --- | --- |
| Message | `msg_` |
| Session | `ses_` |
| Event | `evt_` |
| Request operation | `req_` |
| Replay operation | `rpl_` |
| Trace | `trc_` |
| Client instance | `cli_` |
| Orchestrator instance | `orc_` |

Prefixes are conventions for readability. Receivers MUST NOT rely on prefixes as the only validation mechanism.

## Unknown messages and fields

### Unknown envelope fields

Receivers SHOULD ignore unknown top-level envelope fields if all required fields are valid. Implementations MAY log unknown fields at debug level.

### Unknown `meta` fields

Receivers SHOULD ignore unknown `meta` fields.

### Unknown `kind`

If a receiver gets an unknown `kind`:

- If the message is part of a negotiated capability the receiver does not support, the receiver SHOULD send `error` with code `CAPABILITY_NOT_NEGOTIATED`.
- If the message is advisory and marked optional by the relevant capability, the receiver MAY ignore it.
- Otherwise, the receiver SHOULD send `error` with code `UNKNOWN_MESSAGE_KIND` and MAY close the session after repeated violations.

### Unknown payload fields

Payload schemas decide unknown field behavior. Control-message payloads SHOULD be strict by default. Domain payloads SHOULD follow their domain schema policy.

## Message ordering

Transport message order is not the same as domain event order.

Requirements:

- Receivers MUST use stream `seq` values for durable event ordering.
- Receivers MUST NOT assume two non-event messages are causally ordered unless `correlationId`, `causationId`, `replyTo`, or the relevant message kind defines ordering.
- `event.batch` messages for the same stream MUST be emitted by the orchestrator in ascending durable event order, but clients MUST still validate ranges.

## Message examples

### Minimal empty-payload control message

```json
{
  "protocol": "nerve",
  "version": 1,
  "id": "msg_01JZ7FSRNTW9TS4V4FAVCQWQ57",
  "kind": "ready",
  "ts": "2026-06-26T12:00:02.000Z",
  "data": {}
}
```

### Correlated response

```json
{
  "protocol": "nerve",
  "version": 1,
  "id": "msg_01JZ7FV9EY6W4F16V97SYQ3H6T",
  "kind": "response",
  "ts": "2026-06-26T12:00:03.000Z",
  "replyTo": "msg_01JZ7FV2SHNEHZZ67J6FD865WE",
  "correlationId": "msg_01JZ7FV2SHNEHZZ67J6FD865WE",
  "data": {
    "ok": true,
    "method": "workspace.get",
    "result": {
      "workspaceId": "default"
    }
  }
}
```

### Error response

```json
{
  "protocol": "nerve",
  "version": 1,
  "id": "msg_01JZ7FW1V6YK9R7Y57F7SEVZ6Q",
  "kind": "error",
  "ts": "2026-06-26T12:00:04.000Z",
  "replyTo": "msg_01JZ7FV2SHNEHZZ67J6FD865WE",
  "correlationId": "msg_01JZ7FV2SHNEHZZ67J6FD865WE",
  "data": {
    "code": "METHOD_NOT_FOUND",
    "message": "Unknown protocol method: workspace.get",
    "retryable": false
  }
}
```
